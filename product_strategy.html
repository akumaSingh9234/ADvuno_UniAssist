2 High-Level Problem Analysis

2.1 Performance & Trust

The Problem: The current application suffers from noticeable input lag and slow page loads, particularly during authentication and initial landing.

Root Cause: The underlying framework (Streamlit) re-executes backend scripts on every interaction. It is designed for internal data tools, not consumer-facing products.

Business Impact: In the education sector, performance equals credibility. A laggy interface signals "unreliable," causing high bounce rates among students and parents.

Performance Impact Visualization

2.2 Fragmented User Experience

The Problem: Features function as isolated islands. Data entered in "Diagnostic Tests" does not inform "University Finder."

Root Cause: Lack of a unified state management layer. The architecture treats every module as a separate script.

Business Impact: The promise of an "AI Co-Pilot" is broken. A true co-pilot remembers context; the current system forces users to repeat themselves.

2.3 Data Volatility

The Problem: User progress in long forms (like Profile Pro) is often lost if the browser refreshes or the network fluctuates.

Root Cause: Reliance on ephemeral "Session State" (RAM) instead of persistent database storage.

Business Impact: Critical friction point. Users who lose 20 minutes of work will abandon the platform immediately.

User Drop-off Analysis

3 The Solution: Architecture Strategy

We propose a Hybrid Architecture that decouples the User Interface (Frontend) from the Business Logic (Backend) and the Intelligence Layer (AI).

3.1 Architectural Design: Modular Monolith

Concept: A unified backend codebase that is internally modular (Auth module, Chat module, Payment module).

Why this wins:

Comparison Modular Monolith Why Better
Vs. Microservices Single Deployment Avoids massive operational overhead of managing 20+ separate servers
Vs. Streamlit Monolith Modular Scaling Allows scaling the "AI Brain" independently of the "User Dashboard"

→

→
Data Layer

4 Technology Stack Selection

Component Selected Alternative Why We Chose It
Frontend Next.js (React) Streamlit / Vue.js SEO & Speed: Pages indexed by Google. Instant loads via SSR.
Backend Node.js (Express) Python (Django) Concurrency: Handles thousands of chat connections efficiently.
AI Engine Python (FastAPI) Node.js AI Power: PyTorch, Sentence-Transformers. Isolated from main site.
Vector DB Qdrant (Self-Hosted) Pinecone (Expensive) Cost & Logic: Free on VPS. Excels at Hybrid Search.
Embeddings Self-Hosted Models OpenAI API Privacy: Student docs processed on private server, never sent externally.

6 Strategic Conclusion

This rebuild is not merely a cosmetic upgrade; it is a fundamental shift from a Prototype to a Platform.

Reliability

By moving to Next.js and persistent databases, we eliminate the "lag" and "data loss" that kill user trust.

Intelligence

By connecting features (Diagnostics to Profile to Universities), we deliver on the promise of a true "Co-Pilot."

Scalability

The modular architecture allows us to support 100 or 100,000 students without rewriting the core system.
